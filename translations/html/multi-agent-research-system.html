Claude 现在具备 Research 能力，可以跨网页、Google Workspace 和各类集成进行搜索，以完成复杂任务。

这套多 Agent 系统从原型走到生产的过程，让我们在系统架构、工具设计、提示词工程这几方面学到不少关键经验。所谓多 Agent 系统，是由多个 agent（即在循环中自主使用工具的 LLM）协同工作组成。我们的 Research 功能由一个 agent 根据用户的 query 来规划研究流程，然后使用工具创建若干并行 agent 同时搜索信息。多 Agent 系统会引入 agent 之间的协同、评估和可靠性这些新挑战。

本文拆解我们用下来有效的原则——希望对你构建自己的多 Agent 系统也有用。

多 Agent 系统的好处

研究工作本质上是开放式问题：你很难事先预测需要哪些步骤。复杂话题的探索过程是高度动态、路径依赖的，你没法把固定路径写死。人类做研究时，会随着发现不断调整思路，跟着调查过程中冒出的新线索往下走。

这种不可预测性恰恰让 AI agent 特别适合研究类任务。研究需要随时调头、或顺着旁支去探索的灵活性。模型必须能自主运行多轮，根据中间结果决定下一步走向哪个方向。线性的、一次成型的流水线无法胜任。

搜索的本质是压缩：从巨大的语料里萃取出洞见。子 agent（subagents）让压缩成为可能——它们并行运转、各自拥有独立的上下文窗口，同时探索问题的不同侧面，再把最重要的 token 浓缩后回传给主 agent（lead research agent）。每个子 agent 还提供了关注点分离——独立的工具、prompt 和探索轨迹——从而降低路径依赖、支持彻底且独立的调查。

智能一旦越过某个门槛，多 Agent 系统就成为扩展能力的关键路径。打个比方：过去 10 万年里，单个人类并没有变得指数级聪明，但人类社会在信息时代变得指数级强大，靠的是集体智能和协同能力。即使是通用智能 agent，作为个体也有极限；一群 agent 能做到的事远超个体。

我们的内部评估显示，多 Agent 研究系统特别擅长广度优先的查询——那种需要同时沿多个独立方向并行调研的任务。我们发现：以 Claude Opus 4 作为 lead agent、Claude Sonnet 4 作为子 agent 的多 Agent 系统，在内部研究评估上比单 agent 的 Claude Opus 4 高出 90.2%。例如让它"列出 S&P 500 信息技术板块所有公司的董事会成员"，多 Agent 系统会把任务拆给子 agent 并行处理，最终给出正确答案；而单 agent 系统因为只能慢速串行搜索，根本搜不到。

多 Agent 系统能行得通，主要是因为它们让你舍得花足够的 token 去解决问题。在我们对 BrowseComp（一个测量浏览类 agent 找到难找信息能力的评估）的分析里，三个因素解释了 95% 的性能差异：仅 token 用量本身就解释了 80% 的差异，剩下两个解释因子是工具调用次数和模型选择。这一发现验证了我们的架构方向——把工作分配给拥有独立上下文窗口的多个 agent，给并行推理腾出更多容量。最新版本的 Claude 模型起到了 token 用量的大幅效率倍增器作用：把模型从 Claude Sonnet 3.7 升级到 Claude Sonnet 4 带来的性能提升，比把 Claude Sonnet 3.7 的 token 预算翻倍还大。多 Agent 架构能为单 agent 容量装不下的任务有效扩容 token 使用。

但有代价：这类架构非常烧 token。在我们的数据里，agent 通常比聊天交互多用约 4 倍的 token，多 Agent 系统则比聊天多用约 15 倍的 token。要做到经济上划算，多 Agent 系统需要瞄准任务价值足够高、能负担起这种性能溢价的场景。此外，有些领域要求所有 agent 共享同一上下文，或者 agent 之间依赖很密集，在今天还不适合多 Agent 系统。例如，大多数编程任务可并行的子任务比研究类任务少，而且 LLM agent 还不擅长实时地与其他 agent 协调和委派。我们的体感是：多 Agent 系统在高价值、可大量并行、信息量超过单一上下文窗口、并需要对接大量复杂工具的任务上表现最佳。

Research 系统的架构概览

我们的 Research 系统采用多 Agent 架构 + 编排者-工作者（orchestrator-worker）模式：一个 lead agent 协调整体流程，并把任务派发给若干并行运转的专门子 agent。

当用户提交一条 query 时，lead agent 先分析它、形成策略，然后派生（spawn）出多个子 agent 并行探索不同方面。如上图所示，子 agent 充当智能过滤器——以本例（2025 年的 AI agent 公司）为例，它们迭代地使用搜索工具搜集信息，然后把公司列表回传给 lead agent，由其汇总成最终答案。

传统做法（Retrieval Augmented Generation, RAG）使用静态检索：把和输入 query 最相似的若干 chunk 取回来，用这些 chunk 生成回复。而我们的架构使用多步搜索（multi-step search）——动态地寻找相关信息、随新发现调整、并对结果进行分析以形成高质量答案。

面向 research agent 的提示词工程与评估

多 Agent 系统与单 Agent 系统有关键差异，其中之一是协同复杂度迅速膨胀。我们早期的 agent 出过这些错：为简单 query 派生 50 个子 agent、为不存在的来源在网上无止境地搜、互相用过多更新打扰对方。由于每个 agent 都由 prompt 引导，提示词工程就是我们改进这些行为的主要杠杆。我们总结的一些原则：

  1. 像你的 agent 那样思考。 想迭代 prompt，你必须理解 prompt 的作用。为此我们在 Console 上用系统里完全相同的 prompt 和工具搭建模拟，逐步观察 agent 工作。这立刻让失败模式浮出水面：已有充足结果还继续搜、搜索 query 写得过于冗长、选错工具等等。有效的 prompting 依赖于你对 agent 建立一个准确的心智模型——心智模型对了，最该改的地方就显而易见。
  2. 教编排者如何派活。 在我们的系统里，lead agent 负责把 query 拆成子任务并描述给子 agent。每个子 agent 都需要明确的目标、输出格式、可用工具与信息源的指引、以及清晰的任务边界。任务描述不够详细，agent 就会重复劳动、留下盲区、或找不到必要信息。一开始我们让 lead agent 给简短指令（如"研究一下半导体短缺"），结果发现这种指令含糊到子 agent 误解任务、或与其他 agent 做完全一样的搜索。比如一个子 agent 跑去研究 2021 年汽车芯片危机，另外两个子 agent 各自又在调研 2025 年的供应链——分工完全失效。
  3. 让投入与 query 复杂度匹配。 Agent 难以自己判断该投入多少精力，所以我们把伸缩规则写进 prompt：简单的事实查询只用 1 个 agent + 3-10 次工具调用；直接比较类需要 2-4 个子 agent，每个 10-15 次调用；复杂研究可能用 10+ 个子 agent、职责清晰划分。这些显式指引让 lead agent 合理分配资源，避免对简单 query 过度投入——这是我们早期版本里常见的失败模式。
  4. 工具设计与选择至关重要。 Agent-工具接口和人机界面一样重要。用对工具不仅高效，往往是绝对必要的——比如让 agent 去网上搜只存在于 Slack 里的上下文，注定一开始就要失败。在 MCP 服务器给模型接入外部工具之后，这个问题被放大了：agent 会遇到大量没见过、质量参差不齐的工具描述。我们给 agent 一些显式启发式规则：先检视所有可用工具、把工具用途匹配到用户意图、做广义外部探索时用网页搜索、能用专用工具就别用通用工具。糟糕的工具描述会把 agent 引到完全错误的路径上，所以每个工具都需要有清晰独立的用途和清晰的描述。
  5. 让 agent 自我改进。 我们发现 Claude 4 模型本身就能当出色的 prompt 工程师。给它一个 prompt 和一种失败模式，它能诊断出 agent 为何失败并提出改进。我们甚至专门做了一个工具测试 agent：给它一个有缺陷的 MCP 工具，它会尝试调用并改写工具描述以规避失败。通过反复测试这个工具几十次，这个 agent 发现了关键的细节坑和 bug。这种改进工具人机工效的做法，使后续 agent 用新描述完成任务的耗时下降 40%——因为它们能避开大部分错误。
  6. 先广再窄。 搜索策略应该模仿专家级人类研究者：先扫描全景再钻细节。Agent 默认倾向于写又长又具体的 query，结果返回很少。我们用 prompt 反向纠正：先用简短、宽泛的 query，评估能拿到什么，再逐步收窄聚焦。
  7. 引导思考过程。 扩展思考（Extended thinking）模式让 Claude 在可见的思考过程中输出额外 token，可以作为可控的草稿纸。Lead agent 用 thinking 规划方法、评估哪些工具匹配任务、确定 query 复杂度与子 agent 数量、定义每个子 agent 的角色。我们的测试表明，扩展思考改善了指令遵循、推理和效率。子 agent 也先规划，再在工具结果之后用 交错思考（interleaved thinking）评估结果质量、识别遗漏、调整下一次 query。这让子 agent 更善于适应各种任务。
  8. 并行工具调用大幅改变速度与性能。 复杂研究任务天然要探索很多来源。我们早期的 agent 串行执行搜索，慢得令人发指。为提速，我们引入两层并行：(1) lead agent 并行派生 3-5 个子 agent 而非串行；(2) 子 agent 内部并行调用 3+ 个工具。这两处改动让复杂 query 的研究耗时最多缩短 90%，让 Research 能在几分钟而非几小时内完成更多工作，同时覆盖比其他系统更多的信息。

我们的 prompting 策略注重灌输好的启发式，而不是写死硬规则。我们研究了高水平人类怎么做研究，再把这些策略编码进 prompt——比如把难题拆成小任务、谨慎评估来源质量、根据新信息调整搜索路径、知道何时该深挖（深入研究一个话题）vs 该铺开（并行探索多话题）。我们也主动设置显式护栏来抑制意外副作用，防止 agent 失控。最后，我们专注打造带可观测性和测试用例的快迭代循环。

Agent 的有效评估

构建可靠的 AI 应用离不开好评估，agent 也不例外。但评估多 Agent 系统面临独特挑战。传统评估往往假设 AI 每次都遵循相同步骤：输入 X，系统按路径 Y 产出 Z。但多 Agent 系统不是这样工作的。即使起点相同，agent 也可能走完全不同但都有效的路径——一个可能搜 3 个来源，另一个搜 10 个；它们也可能用不同的工具找到同一个答案。因为我们不一定知道"正确步骤"是什么，所以通常没法用"是否按预设步骤"来评判。我们需要灵活的评估方法——既看 agent 是否达到了正确结果，也看它过程是否合理。

立刻就开始评估，从小样本做起。 Agent 开发的早期，改动的影响通常巨大——因为低垂果子还多。一个 prompt 调整可能把成功率从 30% 拉到 80%。在这种效应规模下，几个测试用例就能看出变化。我们一开始用约 20 个能代表真实使用模式的 query 作为评估集，跑这些 query 通常足以让改动的效果一目了然。我们常听到 AI 团队拖延做评估，理由是"非得有几百用例的大评估才算有用"。最好的做法恰恰相反：先用几条用例做小规模测试立刻起步，而不是等"做完厚实评估"才开始。

LLM-as-judge 评估做得好就能规模化。 研究类输出难以程序化评估——它们是自由文本，很少有唯一正确答案。LLM 天然适合给输出打分。我们用一个 LLM 评审，依据一个评分卡（rubric）对每个输出打分：事实准确性（主张是否与来源一致？）、引用准确性（被引用的来源是否支持主张？）、完整性（要求的方面是否都覆盖到？）、来源质量（是否用了一手来源而非低质二手来源？）、工具效率（是否用对工具、调用次数合理？）。我们试过用多个评审分别评估不同维度，但发现一次 LLM 调用 + 一个 prompt + 输出 0.0–1.0 分数与通过/未通过判断这种方式最稳定、和人类判断最一致。当评估用例有清晰答案（比如"列出研发预算前三的制药公司"）时尤其有效——可以让 LLM 直接判断回答是否正确。用 LLM 做评审让我们可规模化地评估数百个输出。

人类评估能发现自动化遗漏的问题。 实测 agent 的人会找到评估漏掉的边角情况。包括：罕见 query 上的幻觉、系统级故障、隐蔽的来源选择偏差。在我们这里，人工测试员注意到我们早期的 agent 总倾向于优先选择 SEO 内容农场，而忽视权威但排名不高的来源（如学术 PDF、个人博客）。我们在 prompt 里加入来源质量启发式才解决这一问题。即便有了自动化评估，手工测试仍然不可或缺。

多 Agent 系统具有涌现行为（emergent behaviors）——这些行为不是被显式编程出来的。例如，lead agent 的小改动会以不可预测的方式改变子 agent 的行为。成功的关键是理解交互模式，而不只是单个 agent 的行为。因此，针对这些 agent 的最佳 prompt 不是死板的指令，而是协作的框架——定义分工、问题求解方法、以及精力预算。把这件事做对，依赖于细致的 prompt 与工具设计、扎实的启发式、可观测性、紧密的反馈闭环。可以在 我们 Cookbook 的开源 prompt 中看到我们系统的示例 prompt。

生产可靠性与工程挑战

传统软件里，一个 bug 可能让某个功能挂掉、让性能下降、或导致一次故障。在 agentic 系统里，微小的改动会级联放大成巨大的行为变化——给那种需要在长时间运行过程中维护状态的复杂 agent 写代码因此格外困难。

Agent 是有状态的，错误会复合。 Agent 可能长时间运行，跨越大量工具调用维持状态。这意味着我们需要持久化地执行代码、并在过程中妥善处理错误。没有有效的缓解措施，微小的系统故障对 agent 来说可能是灾难性的。出错时不能简单地从头重启：重启代价高、用户体验差。我们因此构建了能从错误发生处恢复的系统。我们也利用模型的智能去优雅地处理问题——例如，让 agent 知道某个工具正在失败、由它自行调整，效果出奇地好。我们把基于 Claude 的 agent 的适应能力，与确定性的安全措施（如重试逻辑、定期 checkpoint）结合在一起。

调试需要新方法。 Agent 做动态决策，即便 prompt 完全一样，跨 run 也是非确定性的，这让调试更难。例如用户反馈说 agent "找不到明显的信息"，但我们看不出原因——是搜索 query 不好？选了差的来源？还是工具失败？我们引入完整的生产追踪（production tracing）之后，就能系统性地诊断为何失败、修复问题。除了标准的可观测性，我们还监控 agent 的决策模式与交互结构——同时不监控具体对话内容以保护用户隐私。这种高层级的可观测性帮我们诊断根因、发现意外行为、修复常见失败。

部署需要精细协同。 Agent 系统是高度有状态、几乎不停运行的 prompt-工具-执行逻辑组成的网。所以每次部署更新时，agent 都可能正处在流程中的任意位置。我们必须防止善意的代码变更打断正在运行的 agent——不能同时把所有 agent 切到新版本。我们采用 彩虹部署（rainbow deployments）：保持新旧两个版本同时运行，逐步把流量从旧版本切到新版本，避免打扰仍在跑的 agent。

同步执行带来瓶颈。 目前，我们的 lead agent 是同步执行 subagent 的——每批 subagent 跑完才能继续。这简化了协调，但在 agent 之间的信息流上形成瓶颈：lead agent 没法对 subagent 进行航向修正，subagent 之间也不能互相协调，整个系统可能因为等一个 subagent 搜完而被卡住。异步执行可以解锁更多并行——agent 并发工作、按需创建新 subagent。但这种异步性会引入结果协调、状态一致性、跨 subagent 的错误传播等新挑战。随着模型能处理更长、更复杂的研究任务，我们相信性能收益会值回这部分复杂度。

总结

构建 AI agent 时，最后一公里往往就是整段旅途的大部分。在开发者机器上能跑的代码库，要变成可靠的生产系统还需要大量工程投入。Agentic 系统中错误的复合性意味着：传统软件里只是小问题的事，在 agent 这里足以让它整个偏航。一步失败，就可能让 agent 走上完全不同的轨迹，导致不可预测的结果。基于本文所述的种种原因，原型与生产之间的鸿沟，往往比预期更宽。

尽管挑战重重，多 Agent 系统在开放式研究任务上已经证明了价值。用户告诉我们，Claude 帮他们找到了原本没想过的商业机会、梳理了复杂的医疗选项、解决了棘手的技术 bug，通过挖掘出自己一个人不会发现的研究关联，省下了数天的工作量。多 Agent 研究系统可以在规模化场景下可靠运行——前提是：细致的工程、全面的测试、对细节苛刻的 prompt 与工具设计、扎实的运维实践，以及对当下 agent 能力有清晰认知的研究、产品、工程团队之间的紧密协作。我们已经看到这些系统在改变人们解决复杂问题的方式。

以下是关于多 Agent 系统的一些零散补充建议。

对跨多轮修改状态的 agent 做 end-state 评估。 评估那些在多轮对话中修改持久化状态的 agent 有独特挑战。和只读式的研究任务不同，每个动作都会改变后续步骤所处的环境，形成依赖关系，传统评估方法很难处理。我们的成功做法是聚焦 end-state 评估而不是逐轮分析——不去判断 agent 是否走了某条特定流程，而是评估它是否到达了正确的最终状态。这种方法承认 agent 可以用不同路径达到同一目标，同时仍然保证它交付了预期结果。对复杂工作流，把评估拆成离散的 checkpoint，在那些应当发生特定状态变化的关键节点验证，而不是试图校验每一个中间步骤。

长时程对话管理（Long-horizon conversation management）。 生产环境中的 agent 经常进行跨数百轮的对话，需要细致的上下文管理策略。对话越拉越长，标准上下文窗口很快就不够用，必须引入智能压缩与记忆机制。我们实现了这样的模式：让 agent 在完成阶段性工作后做总结，把核心信息存到外部记忆，再去做新任务。当上下文逼近上限时，agent 可以派生一个上下文干净的新 subagent，通过细致的交接保持连续性。此外，agent 还能在到达上下文上限时从记忆里取回研究计划等已存信息，避免丢失先前工作。这种分布式做法既防止上下文溢出，又能跨长时间交互保持对话一致性。

子 agent 直接输出到文件系统，避免"传话游戏"。 某些类型的结果让 subagent 直接绕过 lead agent 输出反而能提升保真度和性能。与其逼 subagent 把所有东西都通过 lead agent 传，不如建立产物（artifact）系统：让专门 agent 各自创建可独立持久化的产物。Subagent 调用工具把自己的产出存到外部系统，然后只把轻量引用回传给 coordinator。这能避免多阶段处理中的信息丢失，也能降低把大段输出反复复制到对话历史里产生的 token 开销。这个模式对结构化输出（如代码、报告、数据可视化）特别有效——专用 subagent 的 prompt 写出的结果，往往比让通用 coordinator 过滤一遍要好得多。
