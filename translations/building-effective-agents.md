# 构建高效的 Agent

> **作者**：Erik Schluntz、Barry Zhang（Anthropic）
> **原文**：[Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)
> **发布**：2024-12-19 · **本中文译稿**：2026-05-14
> 本文为社区学习用途的非官方中文翻译，技术名词以括注英文原文为准，最终解释以原文为准。

过去一年里，我们与数十支跨行业的团队合作构建大型语言模型（LLM）Agent。我们持续观察到一个现象：最成功的实现并没有依赖复杂的框架或专用的库，而是用**简单、可组合的模式**搭建出来的。

本文分享我们与客户合作以及自己构建 Agent 过程中沉淀的经验，为开发者提供构建高效 Agent 的实践建议。

## 什么是 Agent？

"Agent" 一词可以有多种定义。一些客户把 Agent 视为完全自主的系统：长时间独立运行、调用各种工具完成复杂任务。另一些人则用这个词指那些按预定义工作流执行的、更"规定动作"式的实现。在 Anthropic，我们把这些变体统称为 **Agentic 系统（agentic systems）**，但在架构上明确区分 **工作流（workflows）** 与 **Agent**：

- **工作流（Workflows）**：通过预定义的代码路径编排 LLM 与工具的系统。
- **Agent**：由 LLM 动态决定自身流程与工具调用、对如何完成任务保有控制权的系统。

下文会详细探讨这两类 agentic 系统。在附录一（"Agent 实战"）中，我们会介绍两个客户特别看重 Agent 价值的领域。

## 何时（以及何时不）使用 Agent

构建基于 LLM 的应用时，我们建议**先找最简单的方案，仅在必要时才提升复杂度**——很多时候根本不需要 agentic 系统。Agentic 系统通常用更高的延迟和成本换取更好的任务表现，你需要判断这种权衡是否值得。

当复杂度确有必要时：**工作流**适合任务定义清晰、要求可预测性与一致性的场景；**Agent** 适合需要大规模灵活性和模型驱动决策的场景。但对许多应用而言，结合检索（retrieval）与上下文示例（in-context examples）优化单次 LLM 调用就已经足够。

## 何时以及如何使用框架

有不少框架可以让 agentic 系统的实现更轻松，包括：

- [Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview)；
- [AWS 推出的 Strands Agents SDK](https://strandsagents.com/latest/)；
- [Rivet](https://rivet.ironcladapp.com/)：一个拖拽式 GUI LLM 工作流构建器；以及
- [Vellum](https://www.vellum.ai/)：另一款用于构建和测试复杂工作流的 GUI 工具。

这些框架封装了调用 LLM、定义和解析工具、串联调用等底层细节，便于上手。但它们往往引入额外的抽象层，让底层 prompt 和响应变得不透明、更难调试，也容易诱使你在简单方案就够用的场景里盲目加复杂度。

我们建议开发者**先直接使用 LLM API**：很多模式只需几行代码就能实现。如果确实要用框架，**请确保你理解底层代码**——对"引擎盖下到底发生了什么"的错误假设是客户出错的常见来源。

可以参考我们的 [cookbook](https://platform.claude.com/cookbook/patterns-agents-basic-workflows) 里的示例实现。

## 构建块、工作流与 Agent

本节梳理我们在生产环境中观察到的 agentic 系统常见模式。我们从最基础的构建块——**增强型 LLM（augmented LLM）**——开始，逐步提升复杂度：从简单的组合式工作流，一直到自主 Agent。

### 构建块：增强型 LLM

agentic 系统的基础构建块是一个被增强过的 LLM，常见增强能力包括**检索、工具、记忆**。我们当前的模型可以主动使用这些能力——自己生成搜索 query、挑选合适的工具、决定要保留哪些信息。

![增强型 LLM](https://www-cdn.anthropic.com/images/4zrzovbb/website/d3083d3f40bb2b6f477901cc9a240738d3dd1371-2401x1000.png)

我们建议聚焦实现层的两件事：把这些能力**针对你的具体场景做定制**，并为 LLM 提供**简洁、文档清晰**的接口。实现方式很多，其中一条路径是借助我们最近发布的 [Model Context Protocol](https://www.anthropic.com/news/model-context-protocol)：开发者只需写一个简单的 [client](https://modelcontextprotocol.io/tutorials/building-a-client#building-mcp-clients)，就能接入快速扩张的第三方工具生态。

本文剩余部分默认每次 LLM 调用都已经具备这些增强能力。

### 工作流：提示链（Prompt Chaining）

提示链把一个任务拆解为一连串步骤，**每次 LLM 调用都处理上一次调用的输出**。你可以在任意中间步骤插入程序化检查（即下图中的 "gate"），确保流程仍在轨道上。

![提示链工作流](https://www-cdn.anthropic.com/images/4zrzovbb/website/7418719e3dab222dccb379b8879e1dc08ad34c78-2401x1000.png)

**何时使用：** 当任务可以干净利落地拆成固定子任务时。主要目的是**用更高的延迟换更高的准确率**——每一次 LLM 调用都变成更简单的子任务。

**适用示例：**

- 先生成营销文案，再翻译成另一种语言。
- 先写文档大纲、校验大纲是否满足某些标准，再基于大纲撰写正文。

### 工作流：路由（Routing）

路由对输入做分类，再把它导向某个**专门的后续任务**。这种工作流可以做到关注点分离、为每类输入构造更专门的 prompt——否则针对一类输入的优化往往会拖累其他输入的表现。

![路由工作流](https://www-cdn.anthropic.com/images/4zrzovbb/website/5c0c0e9fe4def0b584c04d37849941da55e5e71c-2401x1000.png)

**何时使用：** 当任务有可清晰区分的类别、且这些类别更适合分别处理，并且能用 LLM 或传统分类模型/算法**准确分类**时。

**适用示例：**

- 把不同类型的客服请求（一般咨询、退款、技术支持）路由到不同的下游流程、prompt 和工具。
- 把简单/常见的问题路由到更小、更便宜的模型（如 Claude Haiku 4.5），把困难/罕见的问题路由到能力更强的模型（如 Claude Sonnet 4.5），同时优化效果和成本。

### 工作流：并行化（Parallelization）

LLM 有时可以并行处理任务，最终再把各路输出**程序化地合并**起来。这种"并行化"工作流有两种主要变体：

- **切分（Sectioning）**：把一个任务拆成可并行执行的独立子任务。
- **投票（Voting）**：同一任务跑多次，得到多样化的结果再聚合。

![并行化工作流](https://www-cdn.anthropic.com/images/4zrzovbb/website/406bb032ca007fd1624f261af717d70e6ca86286-2401x1000.png)

**何时使用：** 当拆分出的子任务可以并行以加速、或需要多个视角/多次尝试来提升结果置信度时。对于多维度的复杂任务，**每个维度让独立的 LLM 调用专注处理**往往效果更好。

**适用示例：**

- **切分**：
  - 实现 guardrail：一个模型实例处理用户请求，另一个实例同时筛查是否包含不当内容/请求。把"护栏"与"主回复"放在同一次 LLM 调用里通常表现更差。
  - 自动化评估：用多个 LLM 调用分别评估模型在给定 prompt 上的不同方面表现。
- **投票**：
  - 用多个不同的 prompt 评审同一段代码的安全漏洞——任何一个报出问题即视为有风险。
  - 判定一段内容是否不当：让多个 prompt 评估不同维度，或设定不同投票阈值，以在误报和漏报之间取得平衡。

### 工作流：编排者-工作者（Orchestrator-Workers）

在编排者-工作者工作流中，一个**中心 LLM（orchestrator）动态拆分任务**、把子任务派发给若干 worker LLM、然后**汇总它们的结果**。

![编排者-工作者工作流](https://www-cdn.anthropic.com/images/4zrzovbb/website/8985fc683fae4780fb34eab1365ab78c7e51bc8e-2401x1000.png)

**何时使用：** 适合那些**子任务无法事先预知**的复杂任务（例如编程：要改几个文件、每个文件要怎么改，都取决于具体任务）。它在拓扑上看起来和并行化相似，但**关键区别在于灵活性**——子任务不是预先定义的，而是由编排者根据具体输入临时决定。

**适用示例：**

- 每次都要对多个文件做复杂修改的编程产品。
- 需要从多个来源搜集并分析潜在相关信息的搜索任务。

### 工作流：评估者-优化者（Evaluator-Optimizer）

在评估者-优化者工作流中，**一个 LLM 调用负责生成回复，另一个 LLM 调用在循环中提供评估和反馈**。

![评估者-优化者工作流](https://www-cdn.anthropic.com/images/4zrzovbb/website/14f51e6406ccb29e695da48b17017e899a6119c7-2401x1000.png)

**何时使用：** 当我们有清晰的评估标准、且迭代式打磨能带来可衡量的提升时。两个判别信号：第一，当人提出反馈时 LLM 的回复明显能变好；第二，LLM 自己也能产出这样的反馈。这类似于人类作者打磨一份成熟文档时的迭代写作过程。

**适用示例：**

- 文学翻译：译者 LLM 第一次可能抓不住所有微妙之处，但评估者 LLM 能提供有用的批评意见。
- 复杂的搜索任务：需要多轮搜索和分析才能获得全面信息，由评估者判断是否需要继续搜索。

### Agent

随着 LLM 在关键能力上的成熟——理解复杂输入、推理和规划、可靠地使用工具、从错误中恢复——Agent 正在生产环境中崭露头角。Agent 的工作起点通常是用户的一条命令或一次交互式讨论。任务清晰后，**Agent 独立规划和执行**，必要时回到人类身边获取更多信息或判断。执行过程中，让 Agent **在每一步都从环境中获得"ground truth"**（工具调用结果、代码执行结果等）来评估自己的进度至关重要。Agent 可以在检查点处或遇到阻塞时暂停等待人类反馈。任务完成时通常自然终止，但通常也会加上**停止条件**（比如最大迭代次数）以保持控制。

Agent 能处理复杂任务，但实现往往很直白——本质上就是**一个 LLM 在循环中根据环境反馈使用工具**。因此，**精心设计你的工具集和工具文档**至关重要。我们在附录二（"为工具做 Prompt Engineering"）中展开介绍工具开发的最佳实践。

![自主 Agent](https://www-cdn.anthropic.com/images/4zrzovbb/website/58d9f10c985c4eb5d53798dea315f7bb5ab6249e-2401x1000.png)

**何时使用 Agent：** 适合开放式问题——很难甚至无法预测需要多少步、也写不出一条固定的代码路径。LLM 可能会跑很多轮，**你需要对它的决策具备一定信任度**。Agent 的自主性让它特别适合在**可信环境**中扩展任务规模。

但自主性同时意味着更高的成本和**错误累积**的可能。我们建议在沙箱环境中做**充分测试**，并配上合适的护栏（guardrails）。

**适用示例：**

下列示例来自我们自己的实现：

- 一个编码 Agent，用于解决 [SWE-bench 任务](https://www.anthropic.com/research/swe-bench-sonnet)——根据任务描述对多个文件进行编辑；
- 我们的 ["computer use" 参考实现](https://github.com/anthropics/anthropic-quickstarts/tree/main/computer-use-demo)，Claude 操控一台计算机来完成任务。

![编码 Agent 的高层流程](https://www-cdn.anthropic.com/images/4zrzovbb/website/4b9a1f4eb63d5962a6e1746ac26bbc857cf3474f-2400x1666.png)

## 组合与定制这些模式

这些构建块**不是教条**。它们是常见模式，开发者可以根据使用场景重新塑形和组合。成功的关键——和任何 LLM 特性一样——是**度量表现并迭代实现**。再强调一次：**只有当复杂度能明确改善结果时才去增加它**。

## 总结

LLM 领域的成功**不是**建出最复杂的系统，而是建出**最适合**你需求的系统。先从简单的 prompt 开始，用全面的评估来优化，**只有在简单方案显著不足时**才加多步的 agentic 系统。

实现 Agent 时，我们尽量遵循三条核心原则：

1. 在 Agent 设计中保持 **简单（simplicity）**。
2. 通过显式展示 Agent 的规划步骤，优先保证 **透明（transparency）**。
3. 通过细致的工具**文档和测试**，精心打磨你的 **Agent-计算机接口（Agent-Computer Interface，ACI）**。

框架能帮你快速起步，但向生产环境推进时不要犹豫——**拆掉抽象层，用基础组件搭建**。遵循上述原则，你能造出不仅强大、同时**可靠、可维护、被用户信任**的 Agent。

## 附录一：Agent 实战

我们与客户合作的过程中，看到两类对 AI Agent 特别有前景的应用，能很好地展示前面所述模式的实际价值。两类应用都说明：Agent 在那些**同时需要对话与行动、有明确成功标准、能形成反馈闭环、并且能融入有意义的人类监督**的任务中提供的价值最大。

### A. 客户支持（Customer Support）

客户支持把熟悉的 chatbot 界面和工具集成带来的增强能力结合起来。它非常适合做更开放式的 Agent，因为：

- 客服交互天然遵循对话流，同时需要访问外部信息和执行动作；
- 可以集成工具拉取客户数据、订单历史、知识库文章；
- 退款、更新工单这类动作可以由程序代为执行；以及
- 成功率可以通过"用户认定的解决"来清晰度量。

已有多家公司用**按结果计费**的定价模型（只有成功解决才收费）证明了这条路径的可行性，也表明他们对自家 Agent 的效果有信心。

### B. 编码 Agent（Coding Agents）

软件开发领域展现了 LLM 特性的巨大潜力，能力从代码补全演进到**自主问题求解**。Agent 在这里特别有效，因为：

- 代码解决方案可以通过**自动化测试**来验证；
- Agent 可以**用测试结果作为反馈**迭代解决方案；
- 问题空间定义清晰、结构良好；以及
- 输出质量可以被客观度量。

在我们自己的实现里，仅凭一条 pull request 描述，Agent 现在就能解决 [SWE-bench Verified](https://www.anthropic.com/research/swe-bench-sonnet) 基准里的真实 GitHub issue。但需要注意：尽管自动化测试能验证功能正确性，**人工评审仍然不可或缺**——它确保方案符合更广义的系统需求。

## 附录二：为工具做 Prompt Engineering

无论你构建的是哪种 agentic 系统，**工具**很可能是 Agent 的关键组件。[Tools](https://www.anthropic.com/news/tool-use-ga) 通过在我们 API 中指定明确的结构和定义，让 Claude 能与外部服务和 API 交互。当 Claude 决定调用工具时，会在 API 响应中包含一个 [tool use block](https://docs.anthropic.com/en/docs/build-with-claude/tool-use#example-api-response-with-a-tool-use-content-block)。工具的定义和规范应该和你的整体 prompt 一样得到**同等的 prompt engineering 投入**。本附录简要介绍如何为工具做 prompt engineering。

同一个动作往往可以有多种表达方式。例如要描述一次文件修改，你可以让模型写 diff，也可以让它**重写整个文件**；要返回结构化输出，可以用 markdown 也可以用 JSON。在软件工程里，这些差异看似只是"外观"、理论上可以无损相互转换——但对 LLM 来说，**有些格式比另一些难写得多**。写 diff 需要在写新代码之前先在 chunk header 里准确数清楚有多少行在变；写嵌在 JSON 里的代码（相比 markdown）需要额外转义换行符和引号。

我们关于工具格式选择的建议：

- **给模型足够的 token "思考"**，免得把自己写进死胡同。
- **让格式接近模型在互联网文本中天然见过的样子**。
- **避免"格式开销"**——比如要求模型精确数清几千行代码、或者对自己写的代码做字符串转义。

一条经验法则：想想人们在打磨**人机界面（HCI）**上投入了多少精力，就应该在打磨 **Agent-计算机接口（ACI）** 上投入同样多的精力。一些做法建议：

- **把自己代入模型的视角。** 单看描述和参数，是否一眼就能看出这个工具该怎么用，还是需要费力琢磨？如果你需要琢磨，那模型多半也需要。好的工具定义往往包含**使用示例、边界情况、输入格式要求、与其他工具的清晰边界**。
- **改参数名或描述能否让事情更显而易见？** 把它当作给团队里的初级开发者写一份漂亮的 docstring——尤其是当你有许多功能相似的工具时。
- **测试模型如何使用你的工具。** 在我们的 [workbench](https://console.anthropic.com/workbench) 中跑大量样本输入，观察模型会犯什么错误，然后迭代。
- **给工具做[防呆设计（Poka-yoke）](https://en.wikipedia.org/wiki/Poka-yoke)。** 通过调整参数让出错更困难。

我们为 [SWE-bench](https://www.anthropic.com/research/swe-bench-sonnet) 构建 Agent 时，**优化工具花的时间比优化整体 prompt 还多**。比如我们发现：Agent 走出根目录后，使用相对路径的工具就容易出错。修复方案是把工具改成**始终要求绝对路径**——之后模型用这个工具表现得近乎完美。
